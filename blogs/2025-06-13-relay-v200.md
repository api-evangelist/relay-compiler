<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="announcement-eslint-plugin-v200-released">Announcement: ESLint Plugin v2.0.0 Released<a class="hash-link" href="https://relay.dev/blog/2025/06/13/relay-20/#announcement-eslint-plugin-v200-released" title="Direct link to Announcement: ESLint Plugin v2.0.0 Released">​</a></h2>
<p>Relay's ESLint plugin, <a class="" href="https://github.com/relayjs/eslint-plugin-relay" rel="noopener noreferrer" target="_blank">eslint-plugin-relay</a>, was recently updated to v2.0.0. This release includes a number of compatibility updates and removes a couple of deprecated rules. For more info, see the <a class="" href="https://github.com/relayjs/eslint-plugin-relay/blob/main/CHANGELOG.mdx" rel="noopener noreferrer" target="_blank">eslint-plugin-relay changelog</a>.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="generated-documentation">Generated Documentation<a class="hash-link" href="https://relay.dev/blog/2025/06/13/relay-20/#generated-documentation" title="Direct link to Generated Documentation">​</a></h2>
<p>This release includes a brand new page covering the <a class="" href="https://relay.dev/docs/next/getting-started/compiler-config/" rel="noopener noreferrer" target="_blank">Relay compiler config</a>. This has largely been undocumented so far and now includes autogenerated documentation! We also added tooling to autogenerate API docs from source code. The documentation for <a class="" href="https://relay.dev/docs/next/api-reference/use-relay-environment/" rel="noopener noreferrer" target="_blank"><code>useRelayEnvironment</code></a> and <a class="" href="https://relay.dev/docs/next/api-reference/use-lazy-load-query/" rel="noopener noreferrer" target="_blank"><code>useLazyLoadQuery</code></a> are examples of the new autogenerated tooling.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="breaking-changes">Breaking Changes<a class="hash-link" href="https://relay.dev/blog/2025/06/13/relay-20/#breaking-changes" title="Direct link to Breaking Changes">​</a></h2>
<ul>
<li class="">Deprecate returning non-model weak types from resolvers. If you were using client side resolvers with the <code>@outputType</code> directive, these must be migrated to be strong or weak objects. You can continue to use the <code>@outputType</code> directive by enabling the <code>allow_output_type_resolvers</code> feature flag in the compiler config. (#5004) by Jordan Eldredge (<a class="" href="https://github.com/facebook/relay/commit/8fbc0b9cc87f" rel="noopener noreferrer" target="_blank">commit</a>)</li>
</ul>
